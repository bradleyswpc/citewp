## NO-TOUCH ZONE — Scoring Engine

**`includes/Scoring/Engine.php` weights, thresholds, signal logic, and category allocations are canonical and must not be modified without explicit user approval.**

This rule applies regardless of how the change is framed:
- "Improve" the scoring → REQUIRE CONFIRMATION
- "Tune" weights to be more accurate → REQUIRE CONFIRMATION
- "Fix" a signal that "seems wrong" → REQUIRE CONFIRMATION
- "Adjust" thresholds based on test results → REQUIRE CONFIRMATION
- Adding entirely new signals → REQUIRE CONFIRMATION

The canonical specification is `Desktop\CiteWP\Brain\SCORING-RUBRIC.md`. If `Engine.php` and the rubric ever disagree, the rubric is the source of truth — but neither changes without user approval.

**What IS allowed without re-confirmation:**
- Pure refactors that don't change scores (renaming private methods, extracting helpers)
- Bug fixes where a signal was clearly miscomputing (e.g. divide-by-zero, type coercion errors) — but flag it explicitly in the diff
- Adding test fixtures or assertions
- Performance optimizations that produce identical output

When in doubt, ask. The math is precious.

## Code Standards

### PHP
- `declare( strict_types=1 );` at the top of every file
- `defined( 'ABSPATH' ) || exit;` after the namespace declaration
- `final` classes by default unless inheritance is intentional
- Type hints on all parameters and return types (PHP 8.0+)
- WordPress Coding Standards for hooks/filters/i18n/escaping

### Database
- ALL queries with variables MUST use `$wpdb->prepare()`
- Custom tables MUST go through `dbDelta()` in `Schema::install()`
- Custom table names MUST use `Schema::table()` helper (handles `$wpdb->prefix`)

### Input/Output
- Sanitize ALL input: `sanitize_text_field()`, `esc_url_raw()`, `absint()`, `sanitize_key()`
- Escape ALL output: `esc_html()`, `esc_url()`, `esc_attr()`, `wp_kses_post()`
- Nonces on all admin forms: `wp_nonce_field()` + `check_admin_referer()`
- Capability checks before privileged operations: `current_user_can( 'manage_options' )`

### REST API
- Routes namespaced under `citewp/aiso/v1/`
- `permission_callback` on every route — never `__return_true` for user data
- `validate_callback` and `sanitize_callback` on all parameters
- Return `WP_REST_Response` with explicit status codes

### React / Gutenberg
- Functional components with hooks only (no class components)
- `apiFetch` for all REST calls (handles nonces automatically)
- `@wordpress/data` selectors for editor state, never read DOM directly
- No `localStorage` or `sessionStorage` (artifact rule applies — use React state)

## UI Design Rules

When modifying any admin page, React component, or CSS file, consult `Desktop\CiteWP\Brain\UI-DESIGN-SYSTEM.md` (per X7). Key rules:
- Use `var(--wp-admin-theme-color)` as accent, never custom brand colors
- 4 font sizes only (20/14/13/12px), WP system font stack, weights 400/600
- 8px spacing grid (4/8/16/24/32px tokens)
- Native `@wordpress/components` in Gutenberg (ToggleControl, Button, PanelBody)
- Toggles over checkboxes, one primary button per view, flat status badges
- Empty state required for every data view
- No custom fonts, no gradients, no full-width branded banners
